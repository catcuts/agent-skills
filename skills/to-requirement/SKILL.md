# Requirement Miner - 需求智能提炼与挖掘

## 核心能力

此技能帮助用户从任意文章/文档中：
1. **提炼显性需求** - 识别文章明确表达的需求点
2. **挖掘隐性需求** - 多维度扩展可能的需求
3. **输出结构化文档** - 生成层次化的 Markdown 需求文档

## 工作流程

### Step 1: 读取并分析文章

使用 Read 工具读取用户提供的文章文件，理解：
- 文章主题和背景
- 提及的功能/系统/流程
- 目标用户/场景
- 现有痛点或问题

### Step 2: 提炼显性需求

从文章中提取明确表达的需求，分两个层次：

**功能级需求** - 具体的功能点
- 用户可直接操作的功能
- 数据处理能力
- 交互行为

**系统级需求** - 子系统/模块
- 架构层面的模块
- 职责边界明确的子系统
- 可独立运行的服务

### Step 3: 多维度挖掘隐性需求

按照 [references/mining-dimensions.md](references/mining-dimensions.md) 中的框架，从以下维度挖掘：

- **技术实现维度** - 可行性、技术栈、性能要求
- **用户体验维度** - 场景扩展、痛点深挖、体验优化
- **商业价值维度** - 商业模式、市场机会、ROI 分析
- **风险合规维度** - 法律合规、数据安全、潜在风险
- **运维支撑维度** - 监控告警、日志审计、容灾备份
- **扩展性维度** - 未来规划、可扩展接口、模块化设计

### Step 4: 生成需求文档

使用 [assets/requirement-template.md](assets/requirement-template.md) 模板，输出结构化的 Markdown 文档：

```
需求文档-[项目名称]-[日期].md
```

文档结构：
1. **项目概述** - 背景、目标、范围
2. **核心需求** - 显性需求（功能级+系统级）
3. **扩展需求** - 各维度挖掘的隐性需求
4. **需求优先级建议** - P0/P1/P2 分类
5. **后续行动建议** - 下一步工作

## 使用示例

**示例 1: 产品文章分析**
```
用户: 帮我从这篇文章提炼需求：./articles/my-product.md
输出: 需求文档-my-product-2024-01-15.md
```

**示例 2: 技术方案文档**
```
用户: 分析这个技术方案的需求：./docs/system-design.md
输出: 需求文档-system-design-2024-01-15.md
```

## 输出规范

### 需求描述格式

每个需求使用统一格式：

```markdown
### [REQ-XXX] 需求标题

**类型**: 功能级 / 系统级
**来源**: 显性(文章原文) / 隐性(挖掘)
**维度**: 技术实现 / 用户体验 / 商业价值 / 风险合规 / 运维支撑 / 扩展性
**优先级**: P0(核心) / P1(重要) / P2(优化)

**描述**:
需求的具体描述...

**验收标准**:
- [ ] 标准 1
- [ ] 标准 2

**关联需求**: REQ-XXX, REQ-XXX
```

### 优先级定义

- **P0**: 核心需求，MVP 必须包含
- **P1**: 重要需求，显著提升产品价值
- **P2**: 优化需求，锦上添花

## 注意事项

1. **保持客观** - 基于文章内容提炼，避免过度解读
2. **注明来源** - 区分显性需求（文章原文）和隐性需求（挖掘推断）
3. **避免冗余** - 合并相似需求，保持需求列表精简
4. **可执行性** - 确保每个需求都有明确的验收标准
5. **维度覆盖** - 尽量覆盖多个挖掘维度，但不必强行填充
